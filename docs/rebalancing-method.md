# 리밸런싱 방식 설명

이 문서는 `gaps rebalance`가 **현재 보유 포트폴리오를 목표로 어떻게 이동시키는가**(리밸런싱 방식)를 이해 중심으로 설명한다. 구현 관점의 설계 명세는 [`docs/superpowers/specs/2026-06-07-rebalancing-engine-design.md`](superpowers/specs/2026-06-07-rebalancing-engine-design.md)를, 단계별 구현은 [`docs/superpowers/plans/2026-06-07-rebalancing-engine-plan-{a,b,c}.md`](superpowers/plans/)를 참고.

---

## 1. 리밸런싱이란 — `plan`과 무엇이 다른가

| | `gaps plan` (최초 배분) | `gaps rebalance` (리밸런싱) |
|---|---|---|
| 전제 | 보유 없음(현금 10억) | **이미 ETF를 보유 중** |
| 산출 | 전량 신규 매수 계획 | **현재 보유 → 목표로 가는 매수/매도 델타** |
| 핵심 질문 | "무엇을 얼마나 살까" | "**언제**(트리거) 손대고, 보유에서 **무엇을 얼마나 사고팔까**" |

즉 리밸런싱은 두 가지를 푼다.
1. **언제 손대는가** — 트리거 (§4)
2. **어떻게 거래하는가** — 현재 보유와 목표의 차이를 실제 매매 수량으로 (§6)

핵심 원칙: **거래계획 생성은 LLM 0 — 순수 결정론**이라 같은 입력이면 같은 결과(재현 가능)다. LLM은 목표를 다시 만드는 일부 경로(monthly full, reassess 서술)에서만 쓴다.

---

## 2. 전체 흐름 한눈에

```
gaps rebalance {daily|monthly} [--from 직전 산출물 디렉토리]
        │
        ▼
[1] 현재 보유 재평가  (reprice_holdings)
     직전 보유 수량 × 오늘 종가 (+현금) → 현재 비중 + 현재 평가액 V
        │
        ▼
[2] 트리거 평가  (언제 손댈까)
     캘린더(월간) · 드리프트 · 이벤트(VIX 등) · regime 프록시 → 발화 tier 1개
        │ tier == none → 모니터링만, 거래 0, 종료
        ▼
[3] tier별 목표 재산출  (무엇을 목표로)
     monthly      : 전체 파이프라인 재실행 → 새 목표
     daily/event  : 직전 목표 + 결정론 방어 오버레이
     reassess     : macro+risk 재진단 → 위험/안전 비례 조정
        │
        ▼
[4] 거래계획 생성  (어떻게 사고팔까)
     델타 = 목표 − 현재
     → no-trade band(작은 델타 생략, 단 cap 위반 해소는 예외)
     → 정수 수량 변환 + 잔여는 현금 보유
     → turnover(실현 거래액) 계산
     → 실현 비중에 전체 mandate 재검증
        │
        ▼
[5] 산출물 3종 + (발화 시) 슬랙 알림
     (rebalancing).json · _plan.csv · _rationale.md
        │
        ▼
   운영자가 _plan.csv 확인 후 MTS 수동 집행
```

## 3. 입력 — 현재 보유 재평가 (`reprice_holdings`)

리밸런싱하려면 "지금 무엇을 얼마나 들고 있는가"를 먼저 알아야 한다. 시스템은 **직전 산출물**에서 이를 복원한다(추가 입력 0).

1. 직전 `(rebalancing)_plan.csv`(또는 `trade_plan.csv`)에서 **보유 수량**과 **현금 잔액**을 읽는다.
2. 그 수량을 **오늘 종가**(KRX, T+1~2 지연 시 직전 영업일로 walk-back)로 다시 평가한다.
3. 종목별 평가액 + 현금 = **현재 평가액 V**. 각 종목 비중 = 평가액 / V. 현금도 `CASH` 포지션으로 비중에 포함한다(합 = 1.0).

> **중요 — capital이 아니라 V 기준:** 가격이 움직이면 평가액 V는 초기 자본(10억)과 달라진다(수익/손실). 리밸런싱은 *지금의 평가액 V*를 목표 비중으로 재배분하므로, 거래 수량도 V를 기준으로 계산한다. 초기 capital로 계산하면 실제 보유와 어긋나 **MTS 주문 수량이 틀리게 된다**(이 버그는 통합 리뷰에서 잡아 수정됨).

**예시:** 직전에 KODEX200 50주(@10,000원), 현금 0 보유. 오늘 KODEX200이 21,000원으로 올랐다면 → 평가액 V = 50 × 21,000 = 1,050,000원. KODEX200 비중 100%. (capital 1,000,000을 썼다면 잘못된 48주로 계산됐을 것이다 — 실제 보유는 50주.)

## 4. 트리거 — 언제 리밸런싱하는가

"달력이 됐으니까"가 아니라 **목표가 바뀌었거나(국면 변화) 보유가 목표에서 벗어났을 때(드리프트)** 손대는 것이 원칙이다. 네 가지를 한 라우터에서 평가하고, **가장 강한 것 하나**(canonical ladder)로 tier를 정한다.

### 4.1 네 가지 트리거

| 트리거 | 무엇을 보나 | 발화 조건(기본값) | 데이터 |
|---|---|---|---|
| **캘린더(월간)** | 날짜 | 매월 1회 (운영자 `gaps rebalance monthly`) | — |
| **드리프트** | 현재 보유 vs 목표/cap | 단일 ETF > 19%(목표 대비 overweight) · 목표 대비 ±5%p 이탈 · 위험자산 합 > 68% | 현재 보유 재평가 |
| **이벤트** | 시장 충격 | VIX>30 · VKOSPI>25 · 수익률곡선 역전 · KOSPI 급락 · vol 정상화 | yfinance·FRED live fetch |
| **regime 프록시(reassess)** | 국면 변화 신호 | 수익률곡선 전환(spread<−50bp) · vol regime 전환 | FRED·yfinance |

> 임계값(0.19/0.68/±5%p 등)은 **대회 cap(단일 20%, 위험 70%)에서 역산한 안전 버퍼**다. cap에 *닿기 전에* 발화해야 위반(즉시 탈락)을 막는다. 전부 yaml 설정값이라 운영 로그를 보며 조정할 수 있다.

### 4.2 우선순위 (canonical ladder)

같은 날 여러 트리거가 동시에 켜지면 다음 순서에서 가장 높은 하나로 처리한다:

```
event:emergency_defensive  >  monthly  >  reassess
  >  drift:defensive  >  drift:rebalance  >  event:risk_on  >  alert  >  none
```

- 위험 축소(defensive)는 위험 확대(risk_on)보다 항상 우선.
- `none`이면 거래 없이 **모니터링 리포트만** — 불필요한 회전을 막는다.

### 4.3 왜 "정기 주간"이 아니라 "조건부 재진단(reassess)"인가

실측상 매크로 국면(regime)은 주 단위로 거의 안 바뀐다(2026-05~06 3주간 `growth_inflation` 고정). 그래서 **매주 무조건 재배분하는 정기 weekly를 없애고**, 대신 daily 감시가 *국면 변화 신호*(수익률곡선·vol 전환)를 잡았을 때만 macro+risk를 다시 돌려(reassess) 확인한다. 변화가 없으면 거래하지 않는다. "달력이 아니라 상태 변화로" 움직이는 더 원리적인 구조다.

## 5. tier별 목표 재산출 — 무엇을 목표로 삼는가

발화한 tier에 따라 **목표 비중을 만드는 방법**이 다르다.

### 5.1 monthly — 전체 파이프라인 재실행
매월 1회. 분석가 4종 → 디베이트 → allocator 전체를 다시 돌려 **완전히 새 목표**를 만든다. 매크로 지표(CPI·고용 등)가 월간 발표 주기라 월 1회 재진단이 정보 주기와 맞고, 대회 월간 회전율 ≥10%도 이때 보장한다. (LLM 전체 사용)

### 5.2 daily / event — 결정론 방어 오버레이
직전 목표에 **종목 교체 없이 비중만** 조정한다(LLM 0).
- **emergency_defensive** (VIX 급등 등): 위험자산을 방어 목표치(기본 55%)까지 비례 축소하고 안전자산으로 채움.
- **risk_on** (vol 정상화): 위험자산을 소폭 확대(70% cap 내).
- **drift:defensive**: 위험자산을 cap 안으로 축소.
- **drift:rebalance**: 직전 목표로 복원(가격 drift로 벌어진 것을 되돌림).

### 5.3 reassess — 국면 재진단 후 비례 조정
macro+risk를 재실행해 regime이 실제로 바뀌었는지 확인한다.
- **안 바뀌었으면 → 거래 0** (모니터링만).
- 바뀌었으면 → 직전 보유의 **위험/안전 그룹을 비례로만** 조정(예: 위험 −5%p). **종목 집합은 그대로** 두어 교체 비용을 0으로 만든다("보유 우선"의 극단).

## 6. 거래계획 생성 — 어떻게 사고파는가 (핵심)

목표 비중이 정해지면, 현재 보유에서 거기로 가는 **종목별 매수/매도 수량**을 만든다. 전부 결정론.

### 6.1 ① 델타 계산
```
델타[종목] = 목표 비중 − 현재 비중
```

### 6.2 ② no-trade band — 잡소음 거래 억제
`|델타|`가 **50bp(0.005) 미만**이면 거래하지 않고 보유 유지한다. 국내 ETF는 거래비용이 낮지만, 매일 미세한 비중 차이를 따라 거래하면 회전만 늘고 의미가 없다.

> **단, 예외 — cap 위반은 band를 무시한다.** 가격이 올라 어떤 종목이 단일 cap(20%)을 미세하게 넘겼는데(예: 20.3%) 되돌리는 매도가 -0.3%p라서 band에 먹히면, **위반이 그대로 남아 탈락**한다. 그래서 "현재 비중이 hard cap(20%/위험 70%)을 넘었고, 그 매도가 cap 이하로 되돌리는 경우"는 band가 작아도 **반드시 실행**한다.

### 6.3 ③ 정수 수량 + 잔여 현금
```
거래수량 = round(목표금액 / 종가)
목표금액 = 목표비중 × 현재 평가액 V
거래수량 = 목표수량 − 실제 보유수량(prev_qty)   ← 실제 보유에서 차감
```
정수 주(株) 단위로 반올림하므로 약간의 잔돈이 남는다. 이 잔여는 **현금성 ETF를 억지로 사지 않고 그냥 현금으로 보유**한다(사용자 지시). 현금은 다음 리밸런싱 때 포트폴리오의 일부로 다시 인식된다.

### 6.4 ④ turnover (회전율)
```
turnover = (총매수금액 + 총매도금액) / 현재 평가액 V
```
실제 체결될 거래금액 기준이다(band·반올림 반영). monthly는 대회 규칙상 ≥10%를 충족해야 하며, 미달 시 경고한다.

### 6.5 거래계획 예시 (end-to-end)

**상황:** 현재 KODEX200 100% 보유(평가액 V = 100만원, 100주 @10,000원). monthly full이 새 목표 6종 분산을 제시:

| 종목 | 목표 | 현재 | 목표수량 | 현재수량 | 매매구분 | 거래수량 |
|---|---|---|---|---|---|---|
| KODEX200 (위험) | 15% | 100% | 15 | 100 | **SELL** | −85 |
| TIGER코스닥 (위험) | 15% | 0% | 15 | 0 | **BUY** | +15 |
| CD금리 #1 (안전) | 18% | 0% | 18 | 0 | **BUY** | +18 |
| CD금리 #2 (안전) | 18% | 0% | 18 | 0 | **BUY** | +18 |
| CD금리 #3 (안전) | 18% | 0% | 18 | 0 | **BUY** | +18 |
| 단기채 (안전) | 16% | 0% | 16 | 0 | **BUY** | +16 |

→ turnover = (매수 85만 + 매도 85만)/100만 = **170%** (≥10% 충족). 위험자산 30% ≤ 70%, 각 종목 ≤ 20% → mandate 통과.

## 7. mandate 재검증 — 위반 없는지 최종 확인

거래계획대로 거래한 **실제 보유될 비중(realized)** 에 대해 전체 mandate를 다시 검사한다(`validate_rebalance`):

- **단일 ETF ≤ 20%**, **위험자산 ≤ 70%** — 현금은 위험이 아니라 *분모에만* 기여(현금이 많으면 위험 비중이 자연히 낮아짐). 현금을 분모에서 빼고 재정규화하면 cap이 부풀려져 잘못된 위반이 나므로, **현금을 포함한 비중 그대로** 검사한다.
- **상관 클러스터 ≤ 25%** (AI 쏠림 등), **회전율 하한**.

> 검증 대상은 *목표*가 아니라 **no-trade band·반올림까지 반영된 실제 보유될 비중**이다. 그래야 band에 먹혀 남은 위반을 놓치지 않는다. 위반이 남으면 `passed=False`로 명확히 기록하고 경고한다(절대 통과로 오보고하지 않음).

## 9. 운영 — 언제 자동으로 도는가

- **daily 감시는 GitHub Actions cron으로 자동 실행** — 매 한국 영업일(월~금) **07:00 KST**. 이 시각은 ① 전날 밤 **미국장 마감(KST 새벽) 이후**라 코스피에 영향을 주는 VIX·미국 지표가 최신이고, ② 한국장 개장(09:00) 전이라 거래계획을 미리 준비할 수 있다. (내 컴퓨터를 꺼도 GitHub 서버에서 돈다.)
- **트리거 발화 시 슬랙 알림** — `tier`·주요 매매·검증 통과 여부를 보낸다(`SLACK_WEBHOOK_URL` 설정 시).
- **monthly는 수동 실행** — 투자철학 보고서 검토가 필요해 운영자가 의도적으로 돌린다.
- **실제 주문은 항상 사람이 한다** — 시스템은 거래계획(`_plan.csv`)까지만. 운영자가 확인 후 MTS에 수동 입력한다. *"기계가 매일 살펴보고 이상 시 사람을 부른다. 거래 결정·집행은 사람이 한다."*

## 10. 설정값 (yaml `rebalance` 섹션)

```yaml
rebalance:
  drift:
    single_etf_abs_cap: 0.19      # 단일 ETF 드리프트 발화 (cap 0.20 버퍼)
    single_etf_rel_band: 0.05     # 목표 대비 ±5%p 이탈 발화
    risk_asset_abs_cap: 0.68      # 위험자산 드리프트 발화 (cap 0.70 버퍼)
  no_trade_band: 0.005            # 50bp 미만 델타 거래 생략
  turnover_floor_monthly: 0.10    # 월간 회전율 하한
  defensive_target: 0.55          # emergency_defensive 시 위험자산 목표
  reassess_tilt_step: 0.05        # reassess 위험 조정 크기
  reassess_triggers:              # regime 재진단 프록시
    - {name: yield_curve_regime_shift, condition: "spread_10y_2y_bps < -50"}
    - {name: vol_regime_shift, condition: "vix_change_5d > 0.30 OR vix < 18 AND vix_change_5d < -0.30"}
```

> **주의:** 드리프트 발화 버퍼(0.19/0.68)와 거래계획의 band 예외 기준(hard cap 0.20/0.70)은 의도적으로 다르다. 전자는 "cap에 닿기 전 미리 알림", 후자는 "실제 위반 해소"용이다.

---

## 11. 알려진 한계 (후속 개선 대상)

1. **risk_on/reassess의 단일 cap** — 위험 그룹을 비례 확대할 때 개별 ETF가 20%를 넘길 수 있다. 현재는 검증에서 `passed=False`로 *드러내되* 자동 수선은 안 한다(운영자 검토). 자동 repair는 후속.
2. **daily의 상관 클러스터 검증** — daily/reassess는 분석가를 안 돌려 클러스터 정보를 새로 못 얻으므로 클러스터 cap 검증이 생략된다. 종목 교체가 0(비중만 소폭 변경)이라 영향이 작다는 근거로 허용하고 경고를 남긴다. 정밀 재계산은 후속.
3. **monthly 위반 수선/회전율 완화 재시도** 미구현 — monthly 목표는 이미 파이프라인 내부 검증을 통과하므로 실현 단계 위반은 반올림 수준이다.
4. **현재 보유 = 이상적 보유 가정** — 직전 산출물 수량 기준이라 실제 체결 오차·배당·입출금은 미반영. 실계좌 CSV 입력은 후속.
